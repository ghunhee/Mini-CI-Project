# Socket-based Smart Mini CI Server 🚀
**(Linux System Programming)**

요구하신 리눅스 시스템 콜을 적극 활용한 격리된 CI 테스트 환경입니다.

## 🛠 아키텍처 다이렉션 및 핵심 기능

1. **네트워크 및 동시성 (Pthreads & TCP Sockets)**:
   - 서버는 멀티스레드 구조로 동작하며, 여러 클라이언트가 코드를 전송해도 병렬로 처리합니다.
2. **보안 및 격리 (Linux Namespaces Sandboxing)**:
   - `unshare(CLONE_NEWNS | CLONE_NEWPID | CLONE_NEWIPC | CLONE_NEWNET)`를 사용해 리눅스 커널 레벨에서 파일시스템, PID, 네트워크를 완벽히 격리합니다.
   - `mkdtemp()`를 사용하여 스레드마다 충돌 없는 독립적 임시 작업 공간을 할당합니다.
3. **공정한 채점 및 자원 제어 (Timers & Resource)**:
   - 사용자의 코드에 무한 루프가 있을 경우 시스템이 다운되는 것을 막기 위해 `setitimer()`와 `SIGALRM`으로 정확히 2초 타임아웃 강제 종료 로직을 구현했습니다.
   - 억울함 방지를 위해 절대 시간이 아닌 `getrusage()`와 `wait4()`를 이용하여 해당 프로세스 트리가 소모한 순수 User CPU/System CPU Time을 ms 단위로 측정합니다.
4. **에러 가시화 및 Smart Editor Jump**:
   - `pipe()`, `dup2()` 시스템 콜로 gcc의 stderr 로그를 가로채고 파싱하여 한국어 피드백을 제공합니다.
   - 에러 발생 시 클라이언트가 파싱된 줄 번호 데이터를 받아 `execlp("vi", "+줄번호", ...)`를 실행하여 유저가 바로 고칠 수 있도록 돕습니다.

## 💻 컴파일 및 실행 가이드 (리눅스 전용)

이 프로그램은 `unshare` 등 리눅스 특화 시스템 콜을 사용합니다. 반드시 **WSL, Ubuntu 등의 Linux 환경**에서 실행해야 합니다.

### 1. 빌드하기
터미널을 열고 디렉토리로 이동한 뒤 `make`를 입력하세요.
```bash
make
```

### 2. 서버 실행하기
네임스페이스(Sandboxing) 격리 기능을 사용하려면 커널 권한이 필요하므로 `sudo`로 실행해야 합니다.
```bash
sudo ./server
```

### 3. 클라이언트 실행 및 테스트 (다른 터미널 창에서)
서버가 켜진 상태에서 다른 터미널을 열어 테스트 코드를 제출합니다.

**테스트 1: 컴파일 에러 & 에디터 점프 (Smart Error Jump)**
```bash
./client test_error.c
# 에러 메시지와 함께 "에디터(vi)를 열어 즉시 수정하시겠습니까? (y/n)"가 출력됩니다.
# y를 누르면 vi가 켜지며 바로 에러가 난 줄로 커서가 이동합니다!
```

**테스트 2: 무한 루프 타임아웃**
```bash
./client test_inf.c
# 2초 뒤 서버가 타이머로 코드를 강제 종료하고 타임아웃 메시지를 반환합니다.
```

## 학생 제출 시도 횟수 표시

학생 클라이언트는 채점 결과를 받을 때 현재 학번의 누적 제출 횟수를 함께 표시합니다.

```text
Try: #1 | Result: AC    32ms
Try: #2 | Result: WA    28ms
```

- 서버는 학생별 `submit_count`를 관리하고, `ResultHeader.student_submit_count` 필드로 클라이언트에 전달합니다.
- 클라이언트는 상태바 우측에 `Try: #N` 형식으로 현재 제출 회차를 표시합니다.
- 주차 전환 중 기록이 생략되는 예외 상황에서는 `Try: --`로 표시됩니다.
- 이 기능 추가로 프로토콜 버전이 `PROTO_VERSION 3`으로 올라갔습니다.

프로토콜 구조체가 변경되었으므로 서버와 클라이언트는 반드시 같은 `include/protocol.h`를 기준으로 동시에 재빌드해야 합니다.

```bash
make clean
make server_bin client_bin tui_bin
```

## 🤖 AI 및 팀원을 위한 자동 실행 가이드 (AI Agent Execution Guide)

이 섹션은 팀원이나 AI 어시스턴트가 프로젝트의 최신 기능(특히 `Try: #N` 제출 횟수 추적 및 UI 최적화)을 즉시 빌드하고 테스트할 수 있도록 작성된 체크리스트입니다. AI에게 이 `README.md`를 제공하면 아래 절차대로 프로젝트를 오류 없이 실행하고 검증할 수 있습니다.

### Step 1. 전체 프로젝트 빌드 (Build)
최신 프로토콜(`PROTO_VERSION 3`)이 반영되었으므로, 이전 바이너리와의 충돌을 막기 위해 반드시 클린 빌드를 수행해야 합니다.
```bash
make clean
make server_bin client_bin tui_bin
```
> **체크 포인트**: `gcc` 및 `clang` (eBPF 컴파일용)이 설치되어 있어야 하며, 컴파일 에러 없이 바이너리가 생성되어야 합니다.

### Step 2. eBPF 보안 서버 실행 (Server)
서버는 리눅스 커널 수준의 샌드박싱(Namespace 격리)과 eBPF 모니터링을 수행하므로 반드시 루트 권한(`sudo`)이 필요합니다.
새 터미널을 열고 다음 명령을 실행하세요.
```bash
sudo ./server_bin
```
> **체크 포인트**: `[ctx] ServerContext 초기화 완료 — week_1` 로그가 출력되며 서버가 대기 상태에 들어가야 합니다.

### Step 3. 클라이언트 TUI 기반 제출 테스트 (Client)
서버가 켜진 상태에서 **다른 터미널**을 열고 학생 클라이언트를 실행하여 횟수 추적 및 UX를 테스트합니다.
```bash
./client_bin test_01_success.c
```
1. **학번 입력**: 프롬프트가 뜨면 `20240001` 등의 학번을 입력합니다.
2. **1회차 제출 확인**: TUI 화면 우측 상단의 상태바에 `Try: #1 | Result: AC`가 정상적으로 표시되는지 확인합니다.
3. **스마트 재채점 및 누적 카운트 확인 (`Try: #N`)**:
   - TUI 화면에서 `Enter` 키를 눌러 에디터(Vim)로 진입합니다.
   - 코드를 임의로 수정(예: 주석 추가)하고 저장(`:wq`) 후 종료합니다.
   - 터미널에 `[파일이 수정되어 재채점되었습니다...]` 팝업이 뜨고, 엔터를 치면 상태바가 **`Try: #2`** 로 증가하는지 확인합니다.
4. **무의미한 렌더링 방어 및 UX 최적화 확인**: 
   - 다시 `Enter`를 쳐서 에디터에 들어간 뒤, 이번엔 코드를 수정하지 않고 그냥 종료(`:q`)해 봅니다.
   - **재채점을 수행하지 않고, 화면이 깨지는(블랙 스크린) 현상 없이 즉시 메인 TUI 화면으로 매끄럽게 복귀**하는지 검증합니다.
